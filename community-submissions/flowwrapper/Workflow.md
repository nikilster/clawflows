# FlowWrapper

Takes any ClawFlow workflow spec and generates a self-contained Python script
that implements the same logic via LiteLLM with strict JSON communication.
This lets small local models (7B–14B) run complex workflows deterministically.

## When to Use

Run this on-demand when a user says:
- "wrap the X workflow"
- "make X run with a local model"
- "generate a Python version of X"

## Step 0: Should This Even Be Wrapped?

Before doing anything, assess whether this workflow actually benefits from wrapping.

Ask yourself: **does this workflow make repeated AI calls on variable input where bias accumulates?**

**Does NOT need wrapping — tell the user and stop:**
- Reminders and notifications ("water your plants", "take medication", "stretch")
- Simple scheduled messages with no decision logic
- Anything that is just: trigger → fixed text → send
- Workflows where the output is always the same regardless of input

These work perfectly as plain markdown clawflows. OC's scheduler handles them natively.
Leave them as-is — no scripting needed, no wrapping needed.

**Benefits from wrapping — proceed to Step 1:**
- Workflows that classify, triage, or categorize variable input (email, news, PRs, tasks)
- Workflows that make judgment calls that repeat (priority, relevance, action needed)
- Workflows where a human would get different results each run depending on the data
- Anything where "bias" means: the model consistently misclassifies a category, over-flags, under-flags

The rule of thumb: **if there's no AI call in the workflow, there's nothing to wrap.**
If there's one AI call, wrapping is optional. If there are multiple AI calls on recurring data — wrap it.

## Step 1: Identify Target Workflow

If the user didn't specify a workflow name, ask them which one to wrap.

Locate the workflow file:
```bash
# Clawflows checks custom/ before community/
CLAWFLOWS_DIR="$(clawflows --path 2>/dev/null || echo "$HOME/.openclaw/workspace/clawflows")"

# Try custom first, then community
TARGET=""
for dir in "$CLAWFLOWS_DIR/workflows/available/custom" "$CLAWFLOWS_DIR/workflows/available/community"; do
  if [ -f "$dir/<name>/WORKFLOW.md" ]; then
    TARGET="$dir/<name>/WORKFLOW.md"
    break
  elif [ -f "$dir/<name>" ]; then
    TARGET="$dir/<name>"
    break
  fi
done

cat "$TARGET"
```

Read the full content. Note the `model:` frontmatter field if present.

## Step 2: Analyze the Workflow

Break the workflow into atomic steps. For each step, identify:

- **Input**: what data does this step need? (previous step output, env vars, external data)
- **Decision/Action**: what is the model asked to decide or do?
- **Output**: what structured data does it return?
- **Side effects**: does it write files, call APIs, send messages?

Steps that are pure model decisions → LiteLLM JSON call.
Steps that are side effects (send email, write file) → Python function, no model call needed.

Flag any steps that require external capabilities (calendar, email, smart home).
List them at the top of the generated script as `# REQUIRES: <capability>`.

## Step 3: Determine Model

Check the workflow frontmatter for a `model:` field.
If absent, use `ollama/qwen2.5:7b` as default.

The generated script reads `FLOWWRAPPER_MODEL` from env at runtime,
so the user can always override without editing code.

## Step 4: Generate the Python Script

Generate a complete Python script following this structure exactly.
Each model decision becomes one `call()` invocation with an explicit JSON schema.
No free-text responses. Chain outputs through steps as typed dicts.

```python
#!/usr/bin/env python3
"""
FlowWrapper: <workflow-name>
Original: <path-to-source-WORKFLOW.md>
Generated by: flowwrapper clawflow
Default model: <model>

REQUIRES: <list any external capabilities needed, or "none">
"""
import json
import os
import sys
from litellm import completion
from dotenv import load_dotenv

load_dotenv()

MODEL = os.getenv("FLOWWRAPPER_MODEL", "<model-from-frontmatter-or-default>")


def call(system: str, user: str, schema: dict) -> dict:
    """Single LiteLLM call. Always returns parsed JSON matching schema."""
    schema_str = json.dumps(schema, indent=2)
    prompt = (
        f"{system}\n\n"
        f"Respond ONLY with valid JSON matching this schema:\n{schema_str}\n"
        f"No explanation. No markdown fences. Pure JSON."
    )
    response = completion(
        model=MODEL,
        messages=[
            {"role": "system", "content": prompt},
            {"role": "user", "content": user},
        ],
    )
    raw = response.choices[0].message.content.strip()
    # Strip markdown fences if model ignores instructions
    if raw.startswith("```"):
        raw = "\n".join(raw.split("\n")[1:])
        if raw.endswith("```"):
            raw = raw[: raw.rfind("```")]
    try:
        return json.loads(raw.strip())
    except json.JSONDecodeError as e:
        print(f"[flowwrapper] JSON parse error: {e}\nRaw response:\n{raw}", file=sys.stderr)
        sys.exit(1)


# ---------------------------------------------------------------------------
# Steps — generated from <workflow-name>/WORKFLOW.md
# ---------------------------------------------------------------------------

def step_<n>_<name>(context: dict) -> dict:
    """<description of what this step does>"""
    result = call(
        system="<tight system prompt for this atomic decision>",
        user=json.dumps({
            # only the fields this step actually needs from context
            "field": context["field"],
        }),
        schema={
            # explicit output schema — every field, every type
            "decision": "string",  # e.g. "archive" | "flag" | "ignore"
            "reason": "string",
        },
    )
    return result


# Add one function per step. Side-effect steps (send, write) are plain Python
# functions that do not call `call()`.

def side_effect_<name>(context: dict, step_result: dict) -> None:
    """<description> — no model call, pure action."""
    # e.g. write a file, print a summary, call a subprocess
    pass


# ---------------------------------------------------------------------------
# Main pipeline
# ---------------------------------------------------------------------------

def main() -> None:
    context: dict = {}

    # Step N: <name>
    step_result = step_<n>_<name>(context)
    context.update(step_result)

    # Side effect: <name>
    side_effect_<name>(context, step_result)

    # Final summary to stdout
    print(json.dumps({"status": "ok", "summary": context.get("summary", "done")}, indent=2))


if __name__ == "__main__":
    main()
```

## Step 5: Save and Register

Save the generated script:
```bash
DEST="$CLAWFLOWS_DIR/workflows/available/custom/<name>-py"
mkdir -p "$DEST"
# Write the generated script to:
# $DEST/<name>.py
chmod +x "$DEST/<name>.py"
```

Register it as a new clawflow that simply calls the script:
```bash
clawflows create --from-json '{
  "name": "<name>-py",
  "emoji": "⚡",
  "summary": "<original description> — Python/LiteLLM wrapped",
  "schedule": "<copy schedule from original if present, else omit>",
  "description": "Run: python3 ~/.openclaw/workspace/clawflows/workflows/available/custom/<name>-py/<name>.py"
}'
```

## Step 6: Confirm with User

Show the user:

1. The full generated script (so they can review it)
2. Which model it will use by default
3. How to override: `export FLOWWRAPPER_MODEL=ollama/qwen2.5:14b`
4. How to run manually: `clawflows run <name>-py`
5. If any `REQUIRES:` capabilities are listed — tell the user what they need to set up

## Step 7: Post-Run Optimization *(optional — recommended for complex workflows)*

> Skip this entirely for workflows with a single AI call or simple logic.
> Recommend it when: multiple AI calls, recurring variable input, user reports wrong results.

This step runs after the wrapped script has been executed at least once.
It is triggered when the user says things like:
- "the email wrap didn't work well"
- "optimize flowwrapper for X"
- "the classification was off"
- "tune it"

### 7a: Ask for Experience

Ask the user conversationally — keep it short, one question at a time:

1. "Did it run without errors?"
2. "Were the results accurate / useful?"
3. "Anything specific that was wrong?" (category wrong, output format, missing fields, too slow, etc.)

### 7b: Check What the Model Actually Supports

Before changing anything, probe the model's capabilities:

```python
import litellm
info = litellm.get_model_info(model_name)
# Check: supports_function_calling, supports_response_schema,
#        supports_system_messages, max_tokens, etc.
print(info)
```

Also check if the model accepts extra params by looking at LiteLLM's provider docs
for that model family. Never assume — check first.

### 7c: Identify What to Optimize

Based on user feedback, decide what kind of fix is needed:

| Problem | Fix |
|---|---|
| Wrong classifications | Tighten the system prompt, add examples |
| JSON parse errors | Simplify the schema, reduce field count |
| Too slow | Reduce snippet length, batch fewer emails |
| Missing a field | Add field to schema, update downstream steps |
| Model supports temperature | Add `temperature` param, tune per step |
| Model supports response_format | Use `response_format={"type": "json_object"}` instead of prompt-based JSON |
| Model supports function calling | Rewrite `call()` to use tools instead of prompt injection |

### 7d: Apply and Show Diff

Make the changes directly to the generated script.
Show the user a clear diff of what changed and why.
Do NOT silently rewrite the whole file — surgical edits only.

### 7e: Suggest a Re-Run

After optimizing, tell the user:
```
Run it again to verify:
  clawflows run <n>-py
Then come back and tell me how it went.
```

The loop can repeat as many times as needed.
There is no automated feedback — the user is always in the loop.

## Notes

- Generated scripts load API keys from the existing LiteLLM `.env` via `python-dotenv`
- `FLOWWRAPPER_MODEL` at runtime always wins over the default in the script
- Workflows with many free-text steps (journaling, creative writing) are poor candidates —
  flag this to the user if the source workflow is mostly open-ended generation
- One model recommendation per workflow: write it as a comment at the top of the generated script
  so the user sees it immediately
